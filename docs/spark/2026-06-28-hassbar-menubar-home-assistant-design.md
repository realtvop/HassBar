# HassBar Menubar Home Assistant Design

Date: 2026-06-28
Status: Approved design, awaiting written spec review

## Context

HassBar is currently a clean macOS SwiftUI template generated by Xcode 26.3. The app target is a macOS application with a `WindowGroup` entry point, generated Info.plist settings, app sandbox enabled, hardened runtime enabled, and deployment target macOS 15.7. The first product direction is a lightweight macOS menu bar app for viewing Home Assistant entity states and controlling selected devices.

The app should behave as a true menu bar utility: no Dock presence in normal use, a fixed menu bar icon, and settings/entity management opened from the menu bar experience.

## Goals

- Show selected Home Assistant entity states from the macOS menu bar.
- Let the user configure a Home Assistant URL and Long-Lived Access Token.
- Store the token securely in Keychain rather than plain user defaults.
- Load Home Assistant entities, let the user search/filter/select favorites, and show only favorites in the menu.
- Use REST for initial state loading and Home Assistant WebSocket events for realtime state updates.
- Support extensible device controls through Home Assistant service calls, not only switch/light toggles.
- Keep the first version small enough to implement and verify as a single focused app slice.

## Non-Goals

- Home Assistant OAuth/web login is not part of the first version.
- Dynamic menu bar text such as a sensor value or status summary is not part of the first version.
- Multiple Home Assistant instances are not part of the first version.
- Complex service forms or arbitrary service schemas are not part of the first version.
- Notification Center alerts, historical charts, dashboards, and automation editing are not part of the first version.
- Full UI automation testing is not a first-version requirement.

## Recommended Approach

Build a practical MVP around `MenuBarExtra`, manual token configuration, entity selection, and a clear Home Assistant client/store/UI split.

This approach keeps the app true to the menu bar utility shape while still building the important foundation: secure credentials, realtime updates, favorite entities, and domain-based service controls. It also leaves a clean path for OAuth login later because authentication is isolated behind the Home Assistant client boundary.

Alternatives considered:

- A minimal menu-only build would be faster, but it would not fit the selected scope of entity browsing, realtime updates, and extensible control.
- A fully generic Home Assistant console would be more powerful, but too broad for the first version and would work against the lightweight menu bar goal.

## Architecture

The app should be organized into three main layers.

### HomeAssistantClient

Responsible for all Home Assistant communication:

- REST requests to endpoints such as `/api/states` and `/api/services/{domain}/{service}`.
- WebSocket connection to `/api/websocket`.
- Bearer token authentication.
- Request/response decoding.
- Service call payload creation at the transport boundary.

The client should expose focused methods such as:

- Test the configured connection.
- Fetch all entity states.
- Call a Home Assistant service for an entity.
- Start and stop the WebSocket subscription.

The client should not own SwiftUI presentation state.

### HomeAssistantStore

An observable application state object that bridges the client and UI:

- Connection status.
- Full entity cache.
- Favorite entity IDs and ordering.
- Derived favorite entity rows.
- Loading and refresh state.
- Per-entity action state and errors.
- WebSocket reconnect state.

The store coordinates startup loading, manual refresh, entity selection updates, and service call actions.

### SwiftUI Views

Views should remain presentation-focused:

- The app entry point owns the shared store and provides the menu bar scene.
- The menu panel renders connection state and favorite entities.
- Settings views edit HA URL/token and test the connection.
- Entity management views search, filter, select, and order favorites.

Views should call store methods rather than constructing Home Assistant requests directly.

## App Shell

The existing `WindowGroup` template should be converted to a menu bar app shell:

- Use `MenuBarExtra` for the primary entry point.
- Show a fixed system-style icon in the menu bar.
- Hide the Dock icon for normal use with generated Info.plist key `LSUIElement = true`.
- Provide Settings/Preferences and entity management windows from menu actions.
- Keep an explicit Quit action available from the menu.

The menu bar title should not display live sensor values in the first version.

## Persistence

Persist user configuration as follows:

- Home Assistant base URL: `UserDefaults` or `AppStorage`.
- Favorite entity IDs and order: `UserDefaults` or `AppStorage`, encoded as stable IDs.
- Long-Lived Access Token: Keychain.

The app should treat missing URL or missing token as an unconfigured state and guide the user to settings. Tokens should not be logged or shown after saving.

## Home Assistant API Flow

On launch:

1. Load the saved URL, favorite entity list, and token.
2. If configuration is incomplete, show the menu as unconfigured and offer Settings.
3. If configuration is complete, call REST `/api/states`.
4. Populate the entity cache and derive favorite rows.
5. Connect to WebSocket `/api/websocket`.
6. Authenticate over the WebSocket connection.
7. Subscribe to `state_changed` events.
8. Update only the changed entity when an event arrives.

Manual refresh should re-run the REST state fetch and update the cache. WebSocket events remain the preferred path for realtime updates after initial load.

## Entity Selection

The entity management window should use the same entity cache as the menu:

- If no entity cache exists, loading the window triggers a REST fetch.
- If cached entities exist, show them immediately and offer manual refresh.
- Provide search by entity ID and friendly name.
- Provide domain filtering, such as `sensor`, `binary_sensor`, `light`, `switch`, `cover`, `lock`, `scene`, and `script`.
- Let the user select/unselect favorites.
- Preserve favorite ordering.

The menu should show only selected favorite entities to keep the menu compact.

## Menu Panel

The menu panel should be compact and scan-friendly:

- Header: connection status, refresh button, settings button.
- Empty state: configure Home Assistant or select favorite entities.
- Favorite rows: icon, friendly name, current state, unit or important attribute.
- Read-only entities: show state only.
- Controllable entities: show domain-specific action buttons.
- Footer: entity management and quit actions.

Rows should tolerate missing attributes, unavailable entities, and stale data without breaking layout.

## Control Mapping

Device controls should be represented through a small domain-action mapping layer. Each mapping converts a Home Assistant entity into one or more UI actions and then into a service call.

Initial built-in mappings:

- `switch`: `turn_on`, `turn_off`, `toggle`.
- `light`: `turn_on`, `turn_off`, `toggle`.
- `cover`: `open_cover`, `close_cover`, `stop_cover`.
- `lock`: `lock`, `unlock`.
- `scene`: `turn_on`.
- `script`: `turn_on`.

Other domains are read-only in the first version unless a mapping is explicitly added. Service calls should use REST `/api/services/{domain}/{service}` with an `entity_id` payload.

After a successful service call, the app should wait for WebSocket updates or a later refresh to reflect the resulting state. The row may show a brief in-progress state while the call is pending.

## Error Handling

Error handling should be layered so one failure does not make the whole app unusable.

- Missing URL/token: show unconfigured state and a Settings action.
- Invalid URL: show a settings-level validation error.
- Authentication failure: mark the connection as needing updated credentials.
- REST network failure: show offline or refresh failed state; keep last known entity states.
- WebSocket disconnect: show realtime connection lost and attempt limited reconnects.
- Service call failure: show a short per-row error and keep the rest of the menu usable.
- JSON decoding issue for an unexpected entity shape: preserve raw state where possible and avoid crashing.

The app should remain openable, configurable, and quittable even when Home Assistant is unavailable.

## Testing Strategy

Focus first-version tests on units that can be validated without the real Home Assistant server:

- REST request construction: base URL handling, bearer authorization header, endpoint paths.
- REST decoding for Home Assistant state objects.
- Service call payload generation for each supported domain mapping.
- WebSocket `state_changed` event decoding and entity cache update behavior.
- Store behavior for favorite ordering, cache updates, loading state, and error state transitions.
- Keychain wrapper behavior should be isolated behind a small protocol so store logic can use an in-memory fake in tests.

SwiftUI views can start with previews and manual verification. UI automation is optional after the core behavior is stable.

## Implementation Boundaries

The first implementation should avoid broad refactors outside the app's entry point and new feature modules. Existing generated template files can be replaced or split as needed, but unrelated Xcode metadata and user state should be left alone.

Because the template has app sandbox enabled, the app target must include the outgoing network client entitlement so REST and WebSocket calls can reach the user's Home Assistant server.

Suggested module/file grouping:

- `HassBarApp.swift`: app entry and scenes.
- `Models/HAEntityState.swift`: state models and helpers.
- `Services/HomeAssistantClient.swift`: REST/WebSocket transport.
- `Services/KeychainTokenStore.swift`: token persistence boundary.
- `Stores/HomeAssistantStore.swift`: observable app state.
- `Controls/EntityActionMapping.swift`: domain-to-service action mapping.
- `Views/MenuBarView.swift`: menu panel.
- `Views/SettingsView.swift`: connection settings.
- `Views/EntitySelectionView.swift`: favorite selection.

This grouping is a guide, not a requirement. The important boundary is that API, state, action mapping, persistence, and views remain understandable independently.

## Future Extensions

The design intentionally leaves room for:

- OAuth/web login with refresh tokens.
- Dynamic menu bar title for a chosen sensor or summary.
- Light brightness/color controls.
- Service schema discovery and richer service forms.
- Multiple Home Assistant instances.
- Notifications for selected state changes.
- History graphs for sensors.

These should be added only after the first menu bar flow is usable end to end.
